# Briar puppy — two style versions (PixelLab pass)

> **DECISION 2026-06-28 — V2 (simple/compact) is the chosen style.** It reads clean
> at the game's tiny in-world size (~32px); V1's extra detail muddies when shrunk.
> V1 stays in-branch as reference. **Next is a MANUAL pixel-editor pass on V2**
> (crop the dog out of the 124px canvas → snap to the game grid → cleanup → fill /
> mirror the missing side directions for dig/bark → rig walk/run cycles), then build
> the `SpriteFrames` and wire into `scenes/companions/briar.tscn`. Headless agents do
> not do this polish (see the all-caps note below).

Branch `art/briar-puppy-versions`. Generated by `tools/briar_puppy_versions.py` from
the same two Grok concept references, treated as the **same normal puppy in two
looks** (not corrupted, not adult). View = **low top-down** (Projection Canon /
Rule 5), never overridden. **Nothing committed** — review the PNGs in place.

## The two versions

| | Version 1 (chunky) | Version 2 (painterly) |
|---|---|---|
| Reference | `grok-5050ecbe…jpg` (front/left/back/right sheet) | `grok-9889dda1…png`, **front panel A only** |
| Sheet method | clean key + **downscale** of the 4 existing views | **concept-to-sprite** (`create-character-pro`, `create_from_concept`) from the front; PixelLab made the 4 low-top-down rotations |
| Sheet canvas | 64×64 | 124×124 (PixelLab's native pro canvas for size 64; puppy ~64 within it) |
| Sheet dirs | south/north/east/west | south/north/east/west |

**Why V1 isn't `image-to-pixelart`:** the goal said use it "if quality allows" — it
did **not**. On these already-pixel-art views it *degraded* them and hallucinated a
face on the back view, so the higher-quality path was a clean key + downscale of the
originals (preserves the chunky look). See `puppy_v1/params.txt`.

## Animations (PHASE 2) — FIRST-PASS DRAFTS

`puppy_v<1|2>/anim/<name>/<direction>/frame_NN.png`, **5 frames** each, transparent.
Generated via **text-to-animate** (`animate-character`, `mode=v3` action_description).

Animation set = the goal's `idle, walk, run, dig, bark` **plus** Briar's implemented
tells found in `scripts/companions/companion_base.gd`:
- **`seek`** — the alert/point "stare" tell (played in the seek state + on `alert`).
- **`growl`** — directional defensive tell.

(Briar also implements `cower`, `head_bump`, `sit`, `lie_down`, `dusk_press` in the
shipped 32px puppy — **not** re-drafted this pass; out of the goal's listed set.)

### >>> POLISH IS A MANUAL EDITOR STEP — NOT HEADLESS <<<
These are **drafts**: text-to-animate motion is rough at this canvas, and PixelLab's
v3 generator **failed some side directions** (see coverage below). Skeleton rigging,
frame fixes / cleanup, on-grid snapping, and filling the missing directions
(several can be **mirrored** left/right) are a hand pass in the pixel editor. Do not
treat these frames as final.

### Coverage — both versions complete this pass (120 frames each)
- **idle / walk / run / seek / growl** — full 4 directions (south/north/east/west).
- **dig / bark** — south + east (single-pose actions, as implemented in code).
- 24 animation-directions × 5 frames = **120 frames per version**.
- (Some east/west directions needed a second `animate-character` pass — PixelLab's
  v3 generator is flaky per-direction; re-queue is built into the driver.)
